## 1. AI-Accessible Internal Data（AI接続可能な内部データ）

原文: [dora.dev](https://dora.dev/capabilities/ai-accessible-internal-data/)

### 定義

AIシステムを組織のプロプライエタリ情報（コードベース、ドキュメント、運用メトリクス）に安全に接続し、コンテキストアウェアな応答を可能にする。汎用AIモデルを、組織固有のアーキテクチャ・標準・ビジネスロジックに適応した専門家に変換する。

### 核心概念: Prompt Engineering vs Context Engineering

| 概念 | 説明 |
|------|------|
| **Prompt Engineering** | AIへの個別の指示を書く |
| **Context Engineering** | 関連情報（APIドキュメント、社内ポリシー、コードスニペット）を自動収集し、ワークフローを管理する体系的アプローチ |

> 「単一のコマンドではなく、開発者が目的を達成するために必要な完全な『ブリーフィングパケット』を提供すること」

### 研究知見

内部データにアクセスできるAIツールは、**Individual EffectivenessとCode Qualityの統計的に有意な乗数効果**を持つ。

### リスク

- 小バッチを無視するとデリバリー不安定性が増大
- 非推奨パターンの複製による技術的負債
- 不適切なアクセス制御によるデータ漏洩

### 実装の3フェーズ

| フェーズ | 内容 |
|---------|------|
| **Phase 1: 基盤・手動** | プロンプトエンジニアリングからコンテキストエンジニアリングへ。再利用可能なコンテキストテンプレートのバージョン管理ライブラリを作成 |
| **Phase 2: 自動化パイロット** | RAG（Retrieval-Augmented Generation）またはMCP（Model Context Protocol）で1つの高インパクトユースケースを自動化 |
| **Phase 3: スケール・運用化** | リーダーシップスポンサーを確保。堅牢なセキュアな内部APIを構築 |

### よくある落とし穴

| 問題 | 対策 |
|------|------|
| 低品質な内部データ | 単一の高価値データソースでパイロット。AIでデータクリーニング |
| 悪い例がAIを汚染 | ゴールドスタンダードリポジトリをキュレーション。非推奨プロジェクトを除外 |
| コンテキストの腐敗・過負荷 | 量より関連性を重視。RAG/MCPで特定チャンクを取得（context harvesting） |
| セキュリティ・アクセス制御 | 共有サービスアカウントではなくユーザー資格情報による最小権限モデル |

## 2. Documentation Quality（ドキュメント品質）

原文: [dora.dev](https://dora.dev/capabilities/documentation-quality/)

### 定義

内部技術ドキュメントをソフトウェア開発の基本的な構成要素として扱い、明確さ・発見しやすさ・信頼性の8指標で評価する。

### 研究知見

> 「ドキュメント品質は、研究モデルで調査されたすべての技術的プラクティスの実装を駆動する」

平均以上のドキュメント品質を持つチームは、技術ケイパビリティ実装時のパフォーマンス向上が**劇的に増幅**される:

| ケイパビリティ | ドキュメント品質が平均以下 | ドキュメント品質が平均以上 |
|-------------|:------------------:|:------------------:|
| トランクベース開発 | 36% | **1525%** |
| 継続的インテグレーション | 34% | **750%** |
| 継続的デリバリー | 63% | **656%** |
| サプライチェーンセキュリティ | 37% | **451%** |
| SRE | 79% | **343%** |
| 疎結合チーム | 46% | **313%** |
| バージョン管理 | 27% | **278%** |

### 推奨事項

- ドキュメントガイドラインとトレーニングプログラムを確立
- ドキュメントチャンピオンまたはテクニカルライターを指定
- ドキュメントを継続的な組織の仕事として積極的に作成・維持

## 3. Healthy Data Ecosystems（健全なデータエコシステム）

原文: [dora.dev](https://dora.dev/capabilities/healthy-data-ecosystems/)

### 定義

高品質で、アクセスしやすく、統一された内部データで構成される基盤的ケイパビリティ。AI採用の組織的な正の影響を大幅に増幅する。

> "Garbage in, garbage out" — 生成AIの有効性は、入力データの品質とコンテキストに完全に依存する。

### 5つの戦略的原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | **データを戦略的プロダクトとして** | 運用の副産物ではなく、オーナーとスチュワードを指定した管理プロダクト |
| 2 | **統一データソース** | サイロをコンソリデーションまたはフェデレーションで排除。Single source of truth |
| 3 | **自動品質保証** | コードテストを模した継続的バリデーションフレームワーク |
| 4 | **ガバナンス付きアクセス** | ドキュメント化された安全なパスウェイでデータの発見・取得を民主化 |
| 5 | **ドキュメントの同梱** | メタデータとドキュメントをバージョン管理された成果物としてデータ資産に同梱 |

### 測定

- **パーセプション**: データアクセスの容易さ、サイロの頻度、データ品質評価、1時間以内の確定回答到達率
- **客観メトリクス**: データ取得のタイムリネス、データ品質に起因するインシデント頻度、メタデータの鮮度

## 4. Trust in AI（AIへの信頼の醸成）

原文: [dora.dev](https://dora.dev/insights/trust-in-ai/)

### 核心

> 「AIを信頼する開発者ほど生産性が高く、AIをより多く使用する開発者ほどAIを信頼する」

- DORA 2024調査: 75%の回答者がAIによる正の生産性影響を報告
- しかし39%の開発者がAI出力を「少し」または「全く」信頼していない

### 信頼を醸成する5つの戦略

| # | 戦略 | ポイント |
|---|------|---------|
| 1 | **明確な利用規程を策定** | 許可タスク・目的・データ利用を明示。軽量でもポリシーがあれば組織の承認シグナルになる |
| 2 | **コードレビューと自動テストを強化** | セーフガード → 信頼 → AI利用増 → セーフガード強化の好循環。低リスクタスクから開始 |
| 3 | **実体験と習熟の機会を提供** | 慣れが信頼を育てる。好みの言語での実験が特に効果的（評価能力が高まるため） |
| 4 | **奨励するが強制しない** | リーダーの提唱は効果的だが、開発者の自律性（適用レベル・タスク選択）が信頼を高める |
| 5 | **開発者の役割をタスク自動化の先に再定義** | 雇用不安への対処。定型タスク（コーディング、テスト、ドキュメント）からイノベーション・UXへの再配分を構想 |

### 測定

- **サーベイ**: 信頼レベルの知覚評価
- **ログベース**: 提案の受容率、変更リスト提出数、情報検索時間
- **定性**: ポリシーへの認知度、将来の役割への態度

## 5. Version Control（バージョン管理）

原文: [dora.dev](https://dora.dev/capabilities/version-control/)

### 定義

ファイルの作成・アクセス・更新・削除をチーム・組織横断で調整するシステム（Git、Subversion、Mercurial等）。自動化と継続的インテグレーションの基盤。

### AI時代の重要性

> 「AIをワークフローに導入して速く動くが、その速度にはリスクが伴う。バージョン管理はセーフティネットだ。AI生成のミスを即座にリバートでき、監査可能な履歴を提供する。AIのプロンプトと設定を含むすべてをバージョン管理しなければ、ブレーキなしで高速走行しているのと同じだ。」

- 強力なバージョン管理プラクティスは、AI採用がIndividual Effectivenessに与える**正の影響を増幅する**
- AIの高い生成速度と非決定論性のため、頻繁なコミットが「ベストプラクティス」から「クリティカルセーフガード」に昇格

### バージョン管理すべきもの

アプリケーションコード、依存関係、DBスキーマ、環境構築ツール、Dockerfile、テストスクリプト、デプロイ・マイグレーションスクリプト、プロジェクト成果物に加え:

- **AIアーティファクト**（プロンプトとエージェント設定ファイル: GEMINI.md、CLAUDE.md等）

### 測定

- **コミット頻度**: 高頻度ほどIndividual Effectivenessが高い（特にAI利用時）
- **ロールバック依存度**: 頻繁なロールバックがAIのTeam Performance正影響を増幅
- **カバレッジ**: アプリコード、システム設定、ビルドスクリプトのVCS格納率
- **復旧速度**: VCSからのコード復旧・再設定の速度

## 6. Working in Small Batches（小バッチでの作業）

原文: [dora.dev](https://dora.dev/capabilities/working-in-small-batches/)

### 定義

ソフトウェア変更を大規模リリースではなく最小限の増分で開発・テスト・デプロイするプラクティス。迅速なフィードバックループと検証済み学習に基づく素早い軌道修正を可能にする。

### AI時代の重要性

> 「AIは大きなコードブロックを素早く生成することに優れているが、大きな変更はレビュー、テスト、安全な統合が困難である。」

- AI採用が**ソフトウェアデリバリーの不安定性を増加**させる。小バッチは「このリスクへの重要な対策」
- AI採用がProduct Performanceに与える**正の影響を増幅**し、組織的Frictionへの中立的効果を正に転換
- 焦点を「大量生成」から「思慮深い分解と検証」へシフトさせる

### INVEST原則（バッチ設計）

| 原則 | 説明 |
|------|------|
| **Independent** | 任意の順序で完了可能、独立デプロイ可能 |
| **Negotiable** | フィードバックを取り込んで反復可能 |
| **Valuable** | 利用可能で、ステークホルダーに有益 |
| **Estimable** | スコープ推定に十分な情報がある |
| **Small** | スプリント中に数時間〜数日で完了 |
| **Testable** | ユーザー期待に対して検証可能 |

### 実装プラクティス

- 開発者はリリース可能な変更を**少なくとも毎日**トランクにチェックイン
- UI層ではなくサービス/API層から開発を開始
- **ダークローンチ**: ユーザーに見えない状態で機能をデプロイしテスト
- **フィーチャートグル**: 条件付き設定文
- **Branch by abstraction**: 大規模システム変更向け

### よくある落とし穴

| 問題 | 説明 |
|------|------|
| **不十分な分解** | 1週間を超えるコードバッチは大きすぎる。個別機能は最大数日 |
| **リリース前の再結合** | 小バッチをテスト・リリース前にまとめ直すとフィードバックが遅延し、早期欠陥検出を妨げる |
