# AIハーネスによる開発生産性と安全性の統合

> AIが開発に浸透した組織において、「生産性」と「安全性」を対立ではなく共進化として扱うための思索。ハーネスエンジニアリングがこの2軸をどう接続するかを論じる。

> **注記**: 本ドキュメントはファクト（出典付き）と思索（💭マーク）を明確に分離して記述する。

---

## 1. 問題設定：なぜ今「生産性×安全性」なのか

### 1.1 AI生産性パラドックスという症状

2024-2026年のデータは一貫して同じ現象を示している。

| レベル | 指標 | 変化 |
|--------|------|------|
| 個人 | タスク完了数 | +21% |
| 個人 | マージされたPR数 | +98% |
| 組織 | コードレビュー時間 | +91%（悪化） |
| 組織 | PRサイズ | +154%（肥大化） |
| 組織 | バグ率 | +9%（悪化） |
| 組織 | AI支援コードのセキュリティ脆弱性 | +23.7% |

出典: [Faros AI](https://www.faros.ai/blog/ai-software-engineering), [Cortex 2026 Benchmark](https://www.cortex.io/post/ai-is-making-engineering-faster-but-not-better-state-of-ai-benchmark-2026), [Exceeds AI](https://blog.exceeds.ai/industry-benchmarks-ai-code-productivity/)

92.6%の開発者がAIを月次で利用しているにもかかわらず、組織レベルのスループットは10%未満の改善にとどまっている。これはAIの問題ではなく、**組織システムの問題**である。

### 1.2 「速さ」と「安全」の見かけ上の対立

従来、「速く出す」と「安全に出す」はトレードオフとして扱われがちだった。しかし、DORAの10年の実証研究は繰り返し**スピードと安定性は正の相関**であることを示してきた。（→ 証拠の詳細: [DORAの実証: スピードと安定性は正の相関を示す](./evidence/dora-speed-stability-correlation.md)）

AI時代にこの知見はより鋭く問われる。AIはコード生成を高速化するが、その出力の信頼性は人間のコードより低い（AI生成コードのセキュリティ脆弱性は人間の2.74倍 — CodeRabbit調査。また、Veracodeの調査ではAI生成コードの45%がOWASP Top 10脆弱性を含む）。**速度だけを加速すれば、安定性の破壊を加速する**。

💭 **思索**: これが意味するのは、AI時代には「生産性の向上」と「安全性の確保」が同一の基盤（＝ハーネス）によって同時に実現されなければならないということである。生産性向上施策と安全性施策を別々に走らせると、一方が他方を打ち消す構造になる。

## 2. 「消えた生産性」のAI版

### 2.1 広木大地氏の「複雑性のパラドックス」

> 「開発生産効率は恐ろしいスピードで高まっているが、複雑性がそれを上回る」— [広木大地](../perspectives/hiroki-daichi.md)

これはAI以前からの構造的問題である。ツールが偶有的複雑性を削減しても、ソフトウェアが解こうとする問題の本質的複雑性が増大し、体感的な「生産性」は向上しない。

### 2.2 AI版「消えた生産性」— 3つの吸収メカニズム

AI生産性パラドックスの**症状**（個人↑、組織→）は観測されている。では、個人が生み出した余剰はどこへ「消えた」のか。

💭 メカニズムは少なくとも3層に分かれる。

#### メカニズム1: ボトルネック移動（TOCの説明）

AIがコーディング工程を高速化すると、[制約理論（TOC）](../foundations/toc-theory-of-constraints.md)の予測通り、ボトルネックがレビュー・テスト・承認に移動する。Amdahlの法則 — **システムは最も遅い構成要素の速度でしか動かない**。

```
[AI前]  要件定義 ─→ コーディング(制約) ─→ レビュー ─→ デプロイ
[AI後]  要件定義 ─→ コーディング ─→ レビュー(制約) ─→ デプロイ
                     ↑高速化            ↑+91% 待ち時間
```

これは認知されやすく、対処も比較的明確（レビュー自動化、小バッチ規律の徹底等）。

#### メカニズム2: 検証コストの発生（新しい偶有的複雑性）

AIが生成したコードは「潜在的に誤りがある」ものとして扱う必要がある（DORA 2025: 開発者の30%がAI生成コードの信頼性に懸念）。節約した時間は**監査と検証に再配分**される。

- METR研究: 経験豊富な開発者がAI使用で19%遅くなった
- 開発者自身は20%速くなったと認知 → **39ポイントの認知ギャップ**
- AI生成コードのセキュリティ脆弱性は人間の2.74倍（CodeRabbit）、45%がOWASP Top 10脆弱性を含む（Veracode）

出典: [METR 2025](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/), [Exceeds AI](https://blog.exceeds.ai/industry-benchmarks-ai-code-productivity/)

💭 これはブルックスの枠組みで説明できる。AIはコード記述という偶有的複雑性を削減するが、**AI出力の検証という新しい偶有的複雑性**を生み出している。正味の偶有的複雑性削減は、見かけほど大きくない。

#### メカニズム3: 組織の慣性による吸収（最も見えにくい）

💭 上記2つとは質的に異なる第3のメカニズムがある。**組織の慣性がAIで生まれた余剰時間を吸収する**現象である。

- コーディングが速くなった分、**要件が増える**（「今までの工数では無理だった機能」が要求される）
- レビューが高速化されると、**承認プロセスが追加される**（「AIのコードだから二重チェックが必要」）
- 開発者の余裕が生まれると、**会議が増える**（パーキンソンの法則）

これは広木氏の指摘する「営利企業の改善志向と実態の複雑性のバランス」問題のAI版である。余裕が生まれても、組織はその余裕を新たな負荷で埋めてしまう。

**重要な区別**: メカニズム1（ボトルネック移動）とメカニズム2（検証コスト）は技術的に解決可能。メカニズム3（慣性吸収）は組織的・文化的な問題であり、[生成的文化](../foundations/organizational-culture.md)と測定への意識がなければ対処できない。

### 2.3 「消えた生産性」の可視化こそが起点

💭 「消えた生産性」を嘆くのではなく、**どこに消えたかを可視化する**ことがOODAループの「Observe」になる。3つのメカニズムはそれぞれ異なる指標で検出可能であるはず。

| メカニズム | 検出の手がかり | 計測の方向性 |
|-----------|-------------|------------|
| ボトルネック移動 | レビュー待ち時間、キュー長 | リードタイムのステージ分解 |
| 検証コスト | Rework Rate、AI支援PRのインシデント率比較 | AI生成コード品質のトラッキング |
| 慣性吸収 | 要件膨張率、会議時間、開発者のフロー状態劣化 | SPACE的サーベイ + カレンダー分析 |

## 3. 計測の射程 — Layer 1/2/3問題

### 3.1 現在のフレームワークの限界

広木大地氏の[3階層モデル](../perspectives/hiroki-daichi.md)を軸に整理する。

| 階層 | 内容 | 現在の主要フレームワーク | AI時代の問題 |
|------|------|---------------------|------------|
| **Layer 1: 仕事量生産性** | PR数、デプロイ頻度、コード行数 | DORA Four Keys | AIが数値を**容易に膨張**させる。量≠進歩 |
| **Layer 2: 期待付加価値の生産性** | 作った機能の見込み顧客価値 | DX Core 4 (Effectiveness) | AI時代は「間違ったものを速く作る」リスクが増大 |
| **Layer 3: 実現付加価値の生産性** | 実際のビジネス成果 | DX Core 4 (Business Impact) | AIとの因果接続を示すフレームワークなし |

### 3.2 Layer 1の膨張問題

AIの最も直接的な影響はLayer 1にある。PR数+98%、デプロイ頻度増加 — これらはFour Keysを「見かけ上」改善させる。しかし：

- PRサイズが+154%肥大化すれば、デプロイ頻度の上昇は小バッチ規律の崩壊を意味する
- Rework Rateが悪化すれば、デプロイの4割が手戻り修正（[DORA 2025](../research/dora-reports/2025.md)のRework Rate導入はこの問題への回答）
- DORA自身が2025年に4段階分類を廃止し、[7アーキタイプに移行](../research/dora-reports/2025.md#6-7つの組織アーキタイプ)したのは、Layer 1指標だけでは組織の状態を正しく表現できなくなったため

💭 Four Keysは「計器」として重要だが、**AI時代にはそれ単独では「速度計だけで車を運転する」ようなもの**になった。速度は出ているが、正しい方向に進んでいるか（Layer 2）、目的地に到着したか（Layer 3）は別の計器が必要。

### 3.3 Layer 2への接続 — 「正しいものを速く作っているか？」

Layer 2は「期待付加価値」— 作ろうとしている機能がどの程度の顧客価値を持つか。

AIはLayer 2を特殊な形で脅かす。**コーディングコストの劇的な低下が「作るべきかどうか」の意思決定閾値を下げる**。

💭 考えるべき指標の方向性:

| 候補指標 | 何を見るか | 狙い |
|---------|----------|------|
| **Feature adoption rate** | リリースした機能のうち実際に使われた割合 | 「作ったが使われない」機能の検出 |
| **Decision-to-deploy ratio** | 意思決定（要件確定）からデプロイまでの比率と時間 | Layer 1の速度がLayer 2の意思決定品質を犠牲にしていないか |
| **Experiment velocity** | A/Bテスト・仮説検証の回数と学習量 | 「頻度は質に転化する」をLayer 2で実践できているか |
| **Requirements inflation rate** | スプリントあたりの要件追加率 | メカニズム3（慣性吸収）の検出 |

### 3.4 Layer 3への接続 — 「実際にビジネス価値を生んでいるか？」

Layer 3は最も計測困難だが、最も本質的。AIの投資対効果はここでしか評価できない。

💭 考えるべき指標の方向性:

| 候補指標 | 何を見るか | 狙い |
|---------|----------|------|
| **AI leverage ratio** | ビジネス成果 / AI投資（トークン消費量 + ライセンスコスト） | AIへの投資が実際にリターンを生んでいるか |
| **Time-to-value** | 機能リリースから測定可能なビジネス成果までの時間 | デリバリー速度がビジネス価値に変換される速度 |
| **Opportunity cost visibility** | AIで「作れたのに作らなかった」選択肢の追跡 | 慣性吸収されず、意図的に使われた余剰の可視化 |

### 3.5 GAINS Frameworkとの関係

[GAINS Framework](../frameworks/comparison.md)（Faros AI, 2025）はAI固有の10次元測定を提案し、Adoption / Usage / Change Management等を含む。しかし：

💭 GAINSは主にLayer 1のAI拡張版であり、Activity（利用頻度・深度）中心。Layer 2/3への接続は弱い。「AIをどれだけ使っているか」と「AIで何を達成したか」は根本的に異なる問いであり、後者への回答を持つフレームワークはまだ存在しない。

## 4. 安全とは何か — ハーネス文脈での定義

### 4.1 「安全」を定義する前に「何が危険か」を整理する

ハーネスは「AIを御す」ための仕組みであると認識されている。では、御されていないAIが組織にもたらす危険とは何か。

#### 危険の分類（タクソノミー）

💭 AI開発における危険を5つに分類する。

| 危険の類型 | 説明 | 具体例 |
|-----------|------|--------|
| **速度リスク (Velocity Risk)** | 速すぎて制御不能になる | 大量のAI生成コードがレビューをすり抜ける、PRサイズ肥大化による品質低下 |
| **増幅リスク (Amplification Risk)** | 既存の問題をAIが拡大する | 技術的負債の上にAIがコードを積み重ねる、誤った設計パターンの大量複製 |
| **不透明性リスク (Opacity Risk)** | AI判断の根拠が追跡不能 | なぜその実装になったかの理由が残らない、障害時の原因特定困難 |
| **判断力喪失リスク (Displacement Risk)** | 人間が監督に必要な判断力を失う | AIへの過度な依存による開発者スキル低下、ラバースタンプ化するレビュー |
| **系統的リスク (Systemic Risk)** | 複数エージェントの相互作用による創発的リスク | 複数のAIが同時に同じコードベースを変更、矛盾する修正の競合 |

**失敗事例からの教訓**:
- [Amazon AI起因障害（2025年12月〜2026年3月）](./evidence/amazon-ai-outage-2026.md): Kiroマンデート（80%採用義務）下でガードレールなしにAIコードが本番投入 → 12万件+630万件の注文喪失 → **速度リスク + 増幅リスク**の典型。90日間コード安全リセットを発動
- AI生成コードのセキュリティ脆弱性2.74倍 → **不透明性リスク**（脆弱性が入り込んだ理由が不明瞭）

### 4.2 安全の定義 — 4つの性質

💭 上記の危険分類を裏返すと、AI開発における「安全」は以下の4つの性質として定義できる。

#### ① 追跡可能性 (Traceability)

> AIが何をしたか、なぜそうしたかが記録され、事後に検証可能であること。

- 不透明性リスクへの対抗
- ハーネスの機能: 状態管理とトレーサビリティ（意思決定ログ、監査証跡）
- 計測: AI変更のattribution率、障害時のroot cause特定時間

#### ② 可逆性 (Reversibility)

> AIの変更が、いつでも安全に元に戻せること。

- 速度リスクへの対抗
- ハーネスの機能: 制御とガードレール（破壊的操作のブロック、段階的ロールアウト）
- 計測: Failed Deployment Recovery Time、ロールバック成功率

#### ③ 比例性 (Proportionality)

> AIに与える自律性が、リスクの大きさに比例して制御されること。

- 増幅リスク・系統的リスクへの対抗
- ハーネスの機能: コンテキストの段階的開示、権限管理
- 計測: Human-in-the-loop介入率（リスク層別）、自律的操作の範囲と失敗率

#### ④ 人間の判断力維持 (Human Agency)

> 人間がAI出力を意味のある形で評価・判断する能力が維持されていること。

- 判断力喪失リスクへの対抗
- 「ラバースタンプ」ではない、実質的なレビューの確保
- 計測: レビュー差し戻し率の推移（低下は危険信号）、開発者のAIなし作業能力の定期評価

### 4.3 安全と生産性の接続 — 「安全がスループットを生む」

DORAの10年の実証が示す最も重要な知見の一つ:

> **安定性の高いチームは、速度も速い。** セキュリティプラクティスが堅固なチームはバーンアウトが1.4倍少ない（DORA 2022）。生成的文化を持つ組織は30%高い組織パフォーマンス（DORA 2023）。

💭 これをAIハーネスの文脈に翻訳すると:

- **追跡可能性**が高ければ → 障害対応が速くなり → Recovery Timeが短縮される → スループット向上
- **可逆性**が確保されれば → 大胆な変更を恐れなくなり → デプロイ頻度が上がる → 実験速度の向上
- **比例性**が設計されれば → 低リスク操作の自動化が進み → 開発者はLayer 2/3の判断に集中できる
- **人間の判断力**が維持されれば → AI出力の品質向上ループが回り → Rework Rate低下

**安全はスループットの対価ではなく、持続的スループットの前提条件である。**

## 5. OODAループ設計 — AI開発組織のための計測と行動

### 5.1 なぜOODAか

PDCA（Plan-Do-Check-Act）は安定した環境での改善サイクルに適するが、AI開発の変化速度はPDCAの前提を超えている。OODAループ（Observe-Orient-Decide-Act）は、不確実性の高い環境での意思決定ループとして設計されており、AI時代の組織に適合する。

### 5.2 Observe — 何を観測するか

3つのレイヤーと安全性を横断する計測体系。

```
┌─────────────────────────────────────────────────────┐
│                   Observe                            │
│                                                      │
│  Layer 1（仕事量）     Layer 2（期待価値）    Layer 3（実現価値）│
│  ┌──────────┐     ┌──────────┐     ┌──────────┐    │
│  │Four Keys │     │Feature   │     │AI Leverage│    │
│  │+Rework   │     │Adoption  │     │Ratio     │    │
│  │Rate      │     │Rate      │     │          │    │
│  │AI生成比率 │     │Experiment│     │Time-to-  │    │
│  │PRサイズ  │     │Velocity  │     │Value     │    │
│  └──────────┘     └──────────┘     └──────────┘    │
│                                                      │
│  安全性指標                                           │
│  ┌──────────────────────────────────────────┐       │
│  │追跡可能性: AI attribution率               │       │
│  │可逆性: Recovery Time, ロールバック成功率   │       │
│  │比例性: HITL介入率（リスク層別）            │       │
│  │判断力: レビュー差し戻し率の推移            │       │
│  └──────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘
```

### 5.3 Orient — どう解釈するか

観測データを解釈するための2つのレンズ。

#### レンズ1: AI成熟度（The Art of AI Maturity）

組織の現在地を[AI成熟度フレームワーク（Accenture）](./ref-accenture-ai-maturity.md)の17ケイパビリティで評価し、どの能力が不足しているかを特定する。

#### レンズ2: 制約理論（TOC）

「今、何がボトルネックか？」を問う。AI時代のボトルネックは移動する。

```
典型的なボトルネック移動パターン:

Phase 1: コーディングが制約 → AI導入で解消
Phase 2: レビューが制約 → レビュー自動化で緩和
Phase 3: テスト・品質検証が制約 → テスト生成で緩和
Phase 4: 要件定義・設計が制約 → ★ここが本質的複雑性の壁
Phase 5: 組織の意思決定が制約 → ★文化・構造の問題
```

💭 Phase 4以降は技術的ツールでは解決できない。Layer 2/3の生産性が問われる領域であり、ここでのボトルネック解消には組織文化（[Westrum生成的文化](../foundations/organizational-culture.md)）とリーダーシップが必要。

### 5.4 Decide — 何を変えるか

Orientの結果に基づき、ハーネスへの介入を決定する。

| 観測パターン | 解釈 | 介入の方向 |
|-------------|------|----------|
| Layer 1↑ + Layer 2/3→ | AIで量は増えたが価値に転換されていない | Feature adoption計測の導入、要件品質へのフォーカス |
| Rework Rate↑ | 手戻りが増加、品質低下 | ハーネスのfeedbackセンサー強化（テスト生成、構造解析） |
| レビュー差し戻し率↓ | ラバースタンプ化の兆候 | Human-in-the-loopポリシーの見直し、レビュー負荷分散 |
| Recovery Time↑ | AI変更の追跡困難 | トレーサビリティ強化（AI attribution、変更理由の記録） |
| 要件膨張率↑ | 慣性吸収が発生 | 開発者の余剰時間の意図的配分（20%ルール等） |

### 5.5 Act — ハーネスが行動メカニズムである

💭 **ここが最も重要な洞察**: OODAループにおいて、**ハーネスそのものがAct（行動）の実体**である。

従来の開発改善では、Actは「プロセスを変える」「ルールを追加する」「教育する」等の人間の行動変容に依存していた。しかしハーネスは、ソフトウェアの環境設定（AGENTS.md、hooks、権限、プラットフォーム）を変更することで、**AIの振る舞いを直接的に制御**できる。

```
┌──────────────────────────────────────────────────────┐
│ OODAループにおけるハーネスの位置                         │
│                                                       │
│  Observe ──→ Orient ──→ Decide ──→ Act               │
│  (計測)      (解釈)     (判断)     (実行)              │
│    │           │          │         │                 │
│    ↓           ↓          ↓         ↓                 │
│  指標群      TOC/成熟度  介入策    ハーネス変更          │
│  サーベイ    レンズ               ├─ AGENTS.md更新     │
│                                  ├─ hooks追加/変更     │
│                                  ├─ 権限制御変更       │
│                                  ├─ センサー追加       │
│                                  └─ ガイド更新         │
│                                                       │
│  ※ ハーネスなしでは Observe できても Act できない        │
│  ※ 計測だけでは改善しない。計測→環境変更のループが必要   │
└──────────────────────────────────────────────────────┘
```

Martin Fowlerの表現を借りれば:

> 「on the loop」— 個々のAI出力をレビューする（in the loop）のではなく、**アウトプットを生み出すハーネスそのものを改善する**ことで品質を上げる。

出典: [Martin Fowler - Harness Engineering for Coding Agent Users](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)（→ [日本語整理](./ref-fowler-harness-engineering.md)）

これはハーネスの第4機能「継続的改善ループ（Feedback & Recovery）」そのものである:
- ❌ プロンプトを微調整する（個人依存・in the loop）
- ✅ ハーネス（ツール・ルール・センサー）を更新する（組織的・on the loop）

## 6. ハーネス組織のミッションと方向性

### 6.1 The Art of AI Maturityにおける位置づけ

[AI成熟度フレームワーク（Accenture）](./ref-accenture-ai-maturity.md)の5つの成功要因の中で、ハーネスエンジニアリングチームが直接関わるのは以下の2つ。

- **SF03: AIコアの産業化（Data and AI Core）** → 運用可能なデータ・AI基盤の構築。ハーネスはその上位で「AIの出力品質を制御する」レイヤーとして機能
- **SF04: 責任あるAI（Responsible AI）** → 設計段階からの倫理的AI。ハーネスのSensorsとGuidesが実装レベルでResponsible AIを強制

> 💭 **独自解釈**: この2つの成功要因への「またがり」こそがハーネスチームの本質を示している。ハーネスチームは**ガバナンス/コンプライアンスチームではなく、プラットフォームチーム**である。SF03（技術基盤）とSF04（責任あるAI）の交差点に位置し、「安全な速度」を技術的に実現する。

> 💭 **独自解釈**: Accentureのフレームワークは「8ステークホルダー」を特定しているが、ハーネスチームという役割はフレームワーク策定時（2022年）には想定されていない。AI Coding Agent時代に必要な新しいステークホルダーとして位置づける必要がある。

### 6.1.1 ステークホルダーとの相互作用

Accentureの5成功要因に関わる組織機能に対して、ハーネスチームがどう関わるか。

| 組織機能 | 対応する成功要因 | ハーネスチームとの関係 |
|---------|----------------|---------------------|
| **経営層（C-suite）** | SF01: 戦略的リーダーシップ | Layer 1-3の計測結果と安全性レポートを通じたAI投資ROIの可視化 |
| **Platform Engineering** | SF03: AIコアの産業化 | ハーネスの技術基盤（CI/CD統合、hooks、権限制御）を共同構築 |
| **Data Engineering / Analytics** | SF03: AIコアの産業化 | OODAのObserve層（AI生産性指標・安全性指標の収集・分析）を共同構築 |
| **HR / L&D** | SF02: 人材への投資 | AI人材コンピテンシーの技術要件定義をインプット。トレーニング設計に知見を提供 |
| **AI推進委員会** | SF04: 責任あるAI | ハーネスチームは委員会の実行部隊。Responsible AIの実装責任を担う |
| **Corporate IT / Security** | SF04: 責任あるAI | Coding Agentの利用ポリシーとセキュリティポリシーの整合 |

> 💭 **独自解釈**: 重要なのは、ハーネスチームが**全ステークホルダーの「橋渡し」機能**を持つことである。技術（Platform）と文化（HR）とガバナンス（委員会）を、ハーネスという具体的な「製品」を通じて接続する。これはWestrum生成的文化モデルの「橋渡しを奨励する」に対応する。

### 6.2 ミッションの再定義

#### 現在の認識

> 「AIエージェントが安全に最大限のパフォーマンスを発揮できる環境（ハーネス）を設計する」

#### 提案する再定義

💭

> **「AI開発における生産性と安全性の共進化を、計測可能な形で実現するプラットフォームを構築する」**

この再定義のポイント:

| 要素 | なぜ重要か |
|------|----------|
| **共進化** | 生産性と安全性はトレードオフではなく、相互強化の関係にある（DORA実証） |
| **計測可能** | OODAループが回るためには、Observeが機能しなければならない |
| **プラットフォーム** | ガバナンス（制限する側）ではなく、開発者が安全に高速に動けるための基盤 |

### 6.3 方向性 — 3つの柱

#### 柱1: 計測基盤の構築（Observe / Orient）

**Why**: Layer 1指標（Four Keys）だけでは、AI時代の組織の実態を捉えられない。計測なしにはOODAループが回らない。

**What**:
- Layer 1: 既存Four Keys + Rework Rate + AI生成コード比率の計測
- 安全性: AI attribution率、リスク層別HITL介入率の計測
- Layer 2への橋渡し: Feature adoption rateの計測設計（中期）

#### 柱2: ハーネスの標準化と進化（Act）

**Why**: ハーネスなしでは、計測結果を組織的な行動変容に変換できない。個人のプロンプト力に依存する改善は再現性がない。

**What**:
- AGENTS.md / CLAUDE.mdの標準テンプレートと配置義務化
- Martin Fowlerの**Guides（feedforward）とSensors（feedback）の枠組み**に基づくハーネス設計
  - Guides: アーキテクチャドキュメント、コーディング規約、ブートストラップスクリプト
  - Sensors: リンター、構造テスト、レビューエージェント、静的解析
- リスクに比例したHuman-in-the-loopポリシーの設計
- ハーネスの「カバレッジ」評価手法の開発（Fowlerの提起する未解決課題）

#### 柱3: 組織文化の醸成（Orient / 慣性吸収への対抗）

**Why**: メカニズム3（慣性吸収）は技術では解決できない。生成的文化と実験志向がなければ、AIの余剰は無意識に消費される。

**What**:
- AI成熟度フレームワークの**Talent and Culture**領域との連携
- 「消えた生産性」の可視化と対話（開発者サーベイ、レトロスペクティブ）
- AIで生まれた余裕の意図的配分の文化づくり

### 6.4 ハーネスチームの製品 = 「開発者が安全に速く動ける環境」

💭 プラットフォームチームとしてのハーネスチームの「製品」を整理する。

| 製品 | 利用者 | 価値 |
|------|--------|------|
| ハーネステンプレート（AGENTS.md等） | 開発チーム | AIが一貫して安全に動く環境の即時利用 |
| 計測ダッシュボード | 開発チーム・マネジメント | OODAのObserve。データに基づく意思決定 |
| ハーネス設計ガイドライン | 開発チーム | 自チームのハーネスを自律的に改善する能力 |
| AIガバナンスポリシー | 全組織 | リスクに比例した明確なルール。「何が許されるか」の透明性 |
| 安全性ベンチマーク | マネジメント・ステークホルダー | Layer 1-3の生産性指標と安全性指標の統合レポート |

## 7. まとめ — Why / What / How の現在地

### Why: なぜハーネスエンジニアリングが必要か

1. **AI生産性パラドックス**: 個人の速度向上が組織の成果に転換されていない
2. **消えた生産性の3メカニズム**: ボトルネック移動・検証コスト・慣性吸収のいずれもが余剰を吸収する
3. **Layer 1計測の限界**: Four Keysだけでは「速いが無価値」な開発を検出できない
4. **安全と速度の不可分性**: ハーネスなしの速度向上は安定性の破壊を加速する
5. **Act可能性**: ハーネスがOODAループの「行動」を組織的・再現可能にする唯一のメカニズムである

### What: 何をするのか

| 領域 | What | 対応するOODAフェーズ |
|------|------|-------------------|
| **計測基盤** | Layer 1-3の横断的計測体系 + 安全性指標 | Observe |
| **解釈フレーム** | TOC + AI成熟度による制約分析 | Orient |
| **ハーネス標準** | Guides/Sensorsの標準設計、HITL比例ポリシー | Act |
| **文化醸成** | 「消えた生産性」の可視化、余裕の意図的配分 | Orient + Act |

### How: どうやるのか（方向性）

💭 Howは今後のセッションで深堀りすべきだが、方向性として:

- **短期**: Layer 1計測の精緻化（Rework Rate、AI生成コード比率の計測開始）+ AGENTS.md標準化
- **中期**: 安全性指標の設計・導入、ハーネスのGuides/Sensors体系化、Layer 2指標の実験
- **長期**: Layer 3接続（AI leverage ratio等）、ハーネスカバレッジ評価、組織文化変革
