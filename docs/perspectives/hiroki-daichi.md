## 開発生産性に対する考え方

### 生産性の基本定義

- **生産性 = 獲得資源（Output） / 投入資源（Input）**
- ソフトウェア開発ではアウトプットの測定が根本的に困難
- 「最適な単一指標は存在しない」

### 開発生産性の3階層モデル

| 階層 | 説明 | 性質 |
|------|------|------|
| 仕事量生産性 | PR数、コード行数などの作業量 | 先行指標 |
| 期待付加価値の生産性 | 機能に見込まれた顧客価値 | - |
| 実現付加価値の生産性 | 実際に売上に繋がった価値 | 遅行指標 |

### Four Keysの位置づけ

- 「高速で迅速な開発チーム」という定性的目標の定量化
- **「頻度は質に転化する」** が中核の主張
- 継続的デリバリーにより不確実性を環境に曝露させ、フィードバックループで品質向上

### スループットとリードタイムの区別

- **リードタイム**: 開発開始から市場投入までの時間（ビジネス視点）
- **スループット**: 単位時間当たりのアウトプット量（生産性概念）
- 両者にはある程度のトレードオフがある

### 複雑性のパラドックス

- 「開発生産効率は恐ろしいスピードで高まっているが、複雑性がそれを上回る」
- ブルックスの「銀の弾丸はない」を引用し、本質的困難さの4要素を指摘：
  - **複雑性**（非線形に増大）
  - **可変性**（環境変化への対応）
  - **同調性**（他システムとの連携）
  - **不可視性**（抽象的で見えない）

### 単純な測定への警告

- Qiita記事「開発生産性について議論する前に知っておきたいこと」で、単純な測定の危険性を警告
- 営利企業の改善志向と実態の複雑性のバランスを論じている

## 中心テーマ・キーワード

| キーワード | 説明 |
|-----------|------|
| **不確実性** | エンジニアリング組織は不確実性に向き合い、それを削減する営み |
| **心理的安全性** | チームが効果的に機能するための基盤 |
| **技術的負債** | 蓄積された設計上の妥協。「2つのDX」講演で深く論じている |

| **頻度は質に転化する** | 小さく頻繁にデリバリーすることで品質が向上する |
