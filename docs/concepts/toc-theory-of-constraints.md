# 制約理論（TOC）と開発生産性への示唆

エリヤフ・ゴールドラット博士が提唱した制約理論（Theory of Constraints: TOC）の理論体系を整理し、ソフトウェア開発の生産性への示唆を考察する。

出典: [TOC（制約理論）とは | ゴールドラットジャパン](https://www.goldratt.co.jp/about-toc), [制約条件の理論 - Wikipedia](https://ja.wikipedia.org/wiki/%E5%88%B6%E7%B4%84%E6%9D%A1%E4%BB%B6%E3%81%AE%E7%90%86%E8%AB%96)

---

## 1. TOCの基本原理

### 中心的命題

> **「つながり（依存関係）」と「ばらつき（変動性）」のあるシステムには、必ずアウトプットを決定する制約（ボトルネック）が存在する。制約に集中してマネジメントすることで、アウトプットを短期的・飛躍的に向上できる。**

### 3つの前提

1. **依存的事象（Dependency）**: 後続プロセスは先行プロセスに依存する
2. **統計的変動（Variability）**: 個別工程には不可避なばらつきがある
3. **制約の存在**: 上記2つが組み合わさると、システムのアウトプットを決定する制約が必ず生まれる

### 部分最適 vs 全体最適

TOCの最も重要な主張は「**個別工程の効率を上げても、全体のスループットは上がらない**」ということ。制約でない工程をいくら改善しても、全体の産出量は制約工程の能力で決まる。

## 2. スループット会計

従来の原価計算（コストワールド）に対し、TOCはスループットを中心に据えた会計方式（スループットワールド）を提唱する。

### 3つの指標

| 指標 | 定義 | 方向 |
|------|------|------|
| **T（スループット）** | 販売を通じて生み出されるお金の速度。売上から資材費を差し引いたもの | 増大させる |
| **I（在庫/投資）** | 販売しようとするものを購入するために投資したお金。仕掛品・原材料・製品在庫等 | 低減する |
| **OE（業務費用）** | 在庫をスループットに変換するために費やすお金。資材費以外の総経費（直接人件費含む） | 低減する |

### 利益との関係

```
純利益 = T - OE
投資利益率(ROI) = (T - OE) / I
```

### コストワールドとの対比

| 観点 | コストワールド（従来） | スループットワールド（TOC） |
|------|---------------------|--------------------------|
| 優先順位 | コスト削減 > スループット > 在庫 | **スループット増大** > 在庫削減 > コスト削減 |
| 改善対象 | すべての工程を均等に改善 | 制約工程に集中 |
| 部分最適 | 各部門の効率を個別に最大化 | 全体のフローを最大化 |
| 危険性 | コスト削減に限界がある（ゼロ以下にはならない） | スループット増大に理論的上限がない |

## 3. 5つの集中ステップ（継続的改善プロセス）

TOCの根幹をなす改善サイクル。

### ステップ詳細

| # | ステップ | 内容 |
|---|---------|------|
| 1 | **制約を特定する（Identify）** | システムのアウトプットを制限しているボトルネックを見つける |
| 2 | **制約を徹底活用する（Exploit）** | 制約の停止時間を減らし、既存リソースの範囲で生産性を最大化する。追加投資なしでできることをやり切る |
| 3 | **制約以外を従属させる（Subordinate）** | 制約以外の工程を制約のペースに合わせる。非制約工程が制約より速く動いても仕掛品が溜まるだけ |
| 4 | **制約を向上させる（Elevate）** | 設備追加・人員増強等の投資で制約の能力そのものを引き上げる |
| 5 | **惰性を避けて繰り返す（Repeat）** | 制約が解消されたら新たな制約を特定し直す。改善が「当たり前」になって思考停止しないこと |

### 重要な注意

- ステップ2→3→4の**順序が重要**。投資（ステップ4）の前に、まず既存リソースの最大活用（ステップ2）と全体の同期（ステップ3）を行う
- 制約が移動した場合、過去の改善策がかえって害になることがある（ステップ5の「惰性を避ける」）

## 4. DBR（ドラム・バッファー・ロープ）

5つの集中ステップを生産スケジューリングに具体化した手法。

### 3つの構成要素

| 要素 | 役割 | 比喩 |
|------|------|------|
| **ドラム（Drum）** | 制約工程の生産ペースが全体のリズムを決める | 行進のドラム |
| **バッファー（Buffer）** | 制約工程の前に時間的余裕を持たせ、上流の変動を吸収する | 衝撃吸収材 |
| **ロープ（Rope）** | 投入工程を制約工程のペースに連動させ、過剰投入を防ぐ | 先頭と最後尾を結ぶロープ |

### ハイキングの比喩（『ザ・ゴール』より）

子供たちの列でハイキングする場面。最も遅い子（ボトルネック）が全体の速度を決める。

- 遅い子より前の子を速く歩かせても → 列が広がるだけ（仕掛品の増大）
- 遅い子の荷物を他の子に分ける → 全体の速度が上がる（制約の活用・向上）
- 遅い子をロープで先頭と結ぶ → 列の広がりを制限する（WIP制限）

### DBRの3つのパラダイムシフト

| 従来の常識 | TOCの主張 |
|-----------|----------|
| 個別の効率を上げると全体の生産性が上がる | **全体のフローが良くならないと全体の生産性は上がらない** |
| 早く投入すれば早くものができる | **なるべく遅く投入した方が滞留は減り、早くものができる** |
| まとめて作った方が効率は上がる | **まとめて作ると仕掛品が滞留するだけ。一個流しの方がフローは良い** |

## 5. 思考プロセス（Thinking Processes）

TOCが提供する論理的問題解決ツール群。「何を変えるか」「何に変えるか」「どう変えるか」の3つの問いに答える。

| ツール | 目的 | 答える問い |
|--------|------|-----------|
| **現状ツリー（CRT）** | 好ましくない結果（UDE）から根本原因を特定する | 何を変えるか |
| **対立解消図（クラウド/CRD）** | ジレンマ・対立の根底にある仮定を明らかにし、解消する | 何に変えるか |
| **未来ツリー（FRT）** | 提案した解決策が望む結果を生むか検証する | 何に変えるか |
| **ネガティブブランチ（NBR）** | 解決策がもたらす可能性のある悪影響を事前に特定する | 何に変えるか |
| **前提条件ツリー（PRT）** | 実行に必要な前提条件と障害を洗い出す | どう変えるか |
| **移行ツリー（TT）** | 具体的な実行計画をステップバイステップで詳細化する | どう変えるか |

## 6. クリティカルチェーン・プロジェクトマネジメント（CCPM）

TOCをプロジェクト管理に適用した手法。

### 従来のクリティカルパス法との違い

| 観点 | クリティカルパス法 | クリティカルチェーン（CCPM） |
|------|-----------------|--------------------------|
| 考慮する制約 | タスクの依存関係のみ | タスクの依存関係 + **リソースの競合** |
| バッファの扱い | 各タスクに安全余裕を埋め込む | 個別の安全余裕を削り、**プロジェクトバッファ**として末尾に集約 |
| マルチタスク | 許容 | **有害として排除**（コンテキストスイッチによるスループット低下） |
| 進捗管理 | 各タスクの予実管理 | **バッファ消費率**で全体の健全性を管理 |

### 学生症候群とパーキンソンの法則

CCPMが対処する2つの人間行動パターン：

- **学生症候群**: 締め切りまで余裕があると着手を先延ばしにする → 個別タスクの安全余裕が無駄になる
- **パーキンソンの法則**: 仕事は与えられた時間いっぱいまで膨張する → 早期完了分が次工程に渡らない

## 7. ソフトウェア開発生産性への示唆

### TOCとDORAメトリクスの接続

TOCの概念は、DORA 2025レポートが指摘する問題と深く共鳴する。

| TOCの概念 | ソフトウェア開発における対応 |
|-----------|--------------------------|
| 制約に集中せよ | DORAの7アーキタイプは「**何に制約されているか**」で組織を分類する。まさにTOCの制約特定 |
| 部分最適の罠 | AIで個人のコード生成速度（部分）を上げても、レビュー・テスト（制約）が追いつかず全体は改善しない = **AI生産性パラドックス** |
| スループット会計 | DORAがデプロイ頻度よりRework Rateを重視し始めたのは、「量」ではなく「フロー全体の産出」を見る姿勢。コストワールドからスループットワールドへの転換に近い |
| DBRのロープ | **小バッチ規律**（DORA 7つの基盤的能力の一つ）はロープそのもの。投入を制限してフローを維持する |
| 仕掛品の害 | PRサイズ+154%の肥大化は仕掛品の増大。TOCの「まとめて作ると滞留が増えるだけ」に一致 |
| 惰性を避ける | 制約は移動する。AIでコーディングが速くなれば制約はレビューに移る。DORA 2025のリードタイム「**ステージ分解**」の必要性はまさにこれ |

### AI時代のボトルネック移動

```
従来の制約:          コーディング → テスト → デプロイ
                        ↑制約

AI導入後の制約移動:   コーディング → レビュー → テスト → デプロイ
                     (AI加速)      ↑制約が移動
```

AIはコーディング工程の能力を向上させた（ステップ4: Elevate）。しかしステップ5（惰性を避ける）を怠り、制約がレビュー工程に移動したことに気づいていない組織が多い。結果として：

- レビュー時間+91%（新たな制約が顕在化）
- PRサイズ+154%（制約の前に仕掛品が滞留）
- バグ率+9%（制約工程の品質低下）

TOCの観点では、AI導入後に行うべきは「**新たな制約（レビュー・検証工程）を特定し、5つの集中ステップを再開する**」こと。

### CCPMとマルチタスクの問題

CCPMが指摘するマルチタスクの害は、開発者体験にも直結する：

- 開発者の日次タスクコンテキスト+9%、日次PR処理数+47%（DORA 2025データ）
- AI導入でコンテキストスイッチが増加
- Nicole Forsgrenが重視する「**フロー状態**」の阻害要因

### 「早く投入すれば早くできる」の誤り

DBRの「なるべく遅く投入した方が滞留は減り、早くものができる」は、ソフトウェア開発では：

- AIで大量のコードを一度に生成し投入するよりも
- **小さな変更を制約のペースに合わせて投入する方がスループットは高い**
- DORAの「小バッチ規律」の理論的根拠がここにある

---

## 8. まとめ

| TOCの教え | AI時代の開発生産性への示唆 |
|-----------|--------------------------|
| 全体最適 > 部分最適 | 個人の生産性向上より組織のフロー改善 |
| 制約に集中せよ | AIで制約が移動する。新たな制約を特定し直せ |
| 仕掛品を減らせ | PRサイズを小さく。小バッチ規律を維持せよ |
| 投入を制約に同期せよ | レビュー・テストの処理能力に合わせてコード生成量を制御せよ |
| 惰性を避けよ | 「AIで速くなった」で思考停止するな。制約は移動している |
| スループットで測れ | 量（デプロイ頻度）ではなくフロー全体の産出（Rework Rate含む）で評価せよ |
