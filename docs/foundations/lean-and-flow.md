# リーン、フロー効率、WIP制限

## 概要

トヨタ生産方式（TPS）に起源を持つリーン思想は、ソフトウェア開発の生産性を考える上で根本的な理論基盤を提供する。DORAのケイパビリティモデルにおいても、リーン管理の複数の側面がデリバリーパフォーマンスの駆動因子として実証されている。

---

## 1. リーン管理とソフトウェア開発

DORAはリーン管理の以下の側面がソフトウェアデリバリーに影響することを実証した：

- **WIP制限**: 仕掛品の数を制限し、フロー効率を向上
- **作業の可視化**: カンバンボード等による進捗の可視化
- **小バッチ**: 大きな変更を小さな単位に分割
- **バリューストリームの可視化**: 構想からデリバリーまでのフロー全体を可視化

## 2. 継続的デリバリー（Continuous Delivery）

Jez Humbleの著書「Continuous Delivery」（2010年）に基づくプラクティス群。DORAのケイパビリティモデルの中核を構成する。

核心原則：
- **すべての変更をバージョン管理する**（コード、設定、インフラ）
- **ビルド・テスト・デプロイを自動化する**
- **常にリリース可能な状態を維持する**
- **小さなバッチで頻繁にリリースする**
- **フィードバックループを短縮する**

## 3. フロー効率とリソース効率

| 観点 | リソース効率 | フロー効率 |
|------|------------|-----------|
| 最適化対象 | 個人の稼働率 | 作業アイテムのリードタイム |
| 測定 | 「どれだけ忙しいか」 | 「どれだけ速く価値が流れるか」 |
| WIPへの態度 | 多いほど良い（遊休なし） | 少ないほど良い（滞留なし） |
| リスク | 仕掛品の滞留、コンテキストスイッチ | 一時的な遊休時間 |

### リトルの法則

```
リードタイム = WIP / スループット
```

WIPを減らせばリードタイムは短縮される。これはDORAの「変更リードタイム」改善の理論的根拠となっている。

## 4. 小バッチの原則

広木大地氏の「**頻度は質に転化する**」という主張の理論的裏付け：

- 小さなバッチ → 早いフィードバック → 問題の早期検出
- 大きなバッチ → フィードバック遅延 → リスクの蓄積
- DORA 2025レポートで「小バッチ規律」が7つの基盤的能力の一つに選定

### AI時代における小バッチの重要性

AIがコード生成を加速すると、バッチサイズが肥大化する傾向がある（PRサイズ+154%）。TOCの観点では「まとめて作ると滞留が増えるだけ」であり、小バッチ規律の維持がAI時代にはより重要になる。
