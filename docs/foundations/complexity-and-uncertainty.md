# 複雑性と不確実性

## 概要

ソフトウェア開発の生産性を論じる上で避けて通れない根本的な制約：ソフトウェアの本質的な複雑性と、プロジェクトに内在する不確実性。広木大地氏はこれを「不確実性に向き合う」と表現し、エンジニアリング組織論の中核テーマに据えている。

---

## 1. ブルックスの「銀の弾丸はない」

フレデリック・ブルックスが1986年の論文で提唱した、ソフトウェアの本質的困難さの4要素：

| 要素 | 説明 |
|------|------|
| **複雑性（Complexity）** | ソフトウェアの複雑さは非線形に増大する。同じ要素が2つとない |
| **可変性（Conformity）** | 環境変化への対応を求められる。外部インタフェースに合わせ続ける必要がある |
| **同調性（Changeability）** | 他システムとの連携。ソフトウェアは「柔らかい」がゆえに変更圧力を受けやすい |
| **不可視性（Invisibility）** | 抽象的で物理的に見えない。空間的表現を持たない |

### 本質的複雑性 vs 偶有的複雑性

- **本質的複雑性（Essential Complexity）**: 問題領域に内在する複雑さ。取り除けない
- **偶有的複雑性（Accidental Complexity）**: ツール・言語・プロセス等に起因する複雑さ。改善可能

ブルックスの主張：生産性向上のための技術革新は偶有的複雑性しか削減できず、本質的複雑性は残り続ける。

## 2. 広木大地氏の「複雑性のパラドックス」

> 「開発生産効率は恐ろしいスピードで高まっているが、複雑性がそれを上回る」

- 偶有的複雑性を削減するツールは進化し続けている（AI含む）
- しかしソフトウェアが解こうとする問題自体の複雑性も増大している
- 結果として体感的な「生産性」は必ずしも向上しない

### AI時代への示唆

AIはコード生成という偶有的複雑性を大幅に削減するが：
- 本質的複雑性（要件定義、アーキテクチャ判断、ドメイン理解）は削減されない
- むしろAIの出力を検証する新たな偶有的複雑性が生まれている（検証コスト問題）
- METR研究：経験豊富な開発者がAI使用で19%遅くなったのは、検証の認知負荷が原因の可能性

## 3. 不確実性とエンジニアリング組織

広木大地氏『エンジニアリング組織論への招待』の中心テーマ：

> エンジニアリング組織は**不確実性に向き合い、それを削減する営み**である

### 不確実性の種類

| 種類 | 説明 | 対処 |
|------|------|------|
| 方法の不確実性 | どうやって作るかわからない | 技術的スパイク、PoC |
| 目的の不確実性 | 何を作るべきかわからない | 顧客フィードバック、小バッチ |
| 通信の不確実性 | チーム間で認識が揃わない | 心理的安全性、生成的文化 |

### Four Keysとの接続

広木氏は、Four Keysの核心を「**頻度は質に転化する**」と解釈する。

- 継続的デリバリー → 不確実性を市場に曝露 → フィードバック → 不確実性の削減
- デプロイ頻度が高いほどフィードバックループが短くなり、不確実性を早く削減できる
- これはDORAの「スピードと安定性は相関する」という知見の理論的説明になっている

## 4. ブルックスの法則

> 「遅れているソフトウェアプロジェクトに人員を追加すると、さらに遅れる」

- 通信コストは人数の2乗に比例して増大する（n人のチーム → n(n-1)/2 の通信パス）
- 広木大地氏のQiita記事「なぜ、ソフトウェアプロジェクトは人数を増やしても上手くいかないのか」でも論じられている
- DORAの「疎結合チーム」ケイパビリティは、この問題への組織的対処
